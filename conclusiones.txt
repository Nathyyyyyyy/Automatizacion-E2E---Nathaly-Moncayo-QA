1. HALLAZGOS TECNICOS

1.1 ARQUITECTURA Y PATRON UTILIZADO

Se implemento el patron Screenplay de Serenity BDD, que proporciona
una estructura mas legible y mantenible que el tradicional Page Object Model.

El patron Screenplay separa claramente las responsabilidades:
- UI: Elementos de la pagina (Targets)
- Tasks: Acciones de alto nivel que realiza el actor
- Questions: Consultas sobre el estado de la aplicacion
- Tests: Orquestacion del flujo completo

Beneficios:
- Mayor reusabilidad del codigo
- Tests mas expresivos y legibles
- Facilidad para mantener y escalar
- Documentacion viva del comportamiento


1.2 TECNOLOGIAS SELECCIONADAS

Serenity BDD 4.0.21
- Framework de automatizacion con excelente generacion de reportes
- Documentacion viva automática

Screenplay Pattern
- Proporciona abstraccion de alto nivel para describir comportamientos
- Mejora la mantenibilidad del codigo de pruebas

JUnit 5
- Framework de testing moderno
- Mejor soporte para parametrizacion y extensiones

WebDriverManager
- Gestion automática de drivers de navegador
- Elimina configuracion manual de ChromeDriver

Maven 3.9.1+
- Gestion de dependencias
- Ciclo de vida del proyecto bien definido

Chrome Browser
- Navegador principal para las pruebas
- Driver descargado automaticamente

- Validacion visual del flujo de prueba

2. CONFIGURACION Y OPTIMIZACIONES IMPLEMENTADAS

2.1 OPCIONES DE CHROME CONFIGURADAS

Se implementaron opciones de Chrome para mejorar la automatizacion:

- start-maximized: Abre el navegador maximizado
- disable-infobars: Desactiva barras de informacion
- disable-popup-blocking: Desactiva bloqueo de popups
- disable-notifications: Desactiva notificaciones
- disable-dev-shm-usage: Mejora en sistemas con recursos limitados
- no-sandbox: Mejora de compatibilidad
- ignore-certificate-errors: Ignorar errores de certificados

Prefs deshabilitadas:
- credentials_enable_service: false (desactiva guardado de contraseñas)
- profile.password_manager_enabled: false (desactiva manager de contraseñas)
- password_manager_leak_detection_enabled: false (desactiva alerta de brechas)

Estas configuraciones eliminan prompts interruptivos que afectan la
automatizacion sin supervisión.


2.2 MODOS DE EJECUCION

Modo Normal (GUI)
- Ejecuta: mvn clean verify
- El navegador es visible durante la ejecucion
- Util para debugging y visualizacion del flujo

Modo Headless
- Ejecuta: mvn clean verify -Denvironments=headless
- El navegador se ejecuta sin interfaz grafica
- Util para CI/CD y ejecucion en servidores sin display


2.3 SCRIPTS .bat (SIMPLIFICACION DE COMANDOS)

El proyecto incluye scripts .bat para simplificar ejecucion en Windows:

descargar-dependencias.bat
- Equivale a: mvn clean install -DskipTests
- Funcion: Descargar e instalar dependencias
- Uso: Doble clic para ejecutar automaticamente

ejecutar-pruebas.bat
- Equivale a: mvn clean verify
- Funcion: Ejecutar todas las pruebas
- Uso: Doble clic para ejecutar automaticamente

ejecutar-headless.bat
- Equivale a: mvn clean verify -Denvironments=headless
- Funcion: Ejecutar sin interfaz grafica
- Uso: Doble clic para ejecutar automaticamente

ejecutar-prueba-especifica.bat
- Proposito: Ejecutar una prueba especifica
- Uso: Modificar el script con el nombre de la prueba deseada

NOTA IMPORTANTE:
Los scripts .bat son opcionales y solo ofrecen conveniencia en Windows.
El comando base que siempre funciona es: mvn clean verify
Este comando ejecuta la prueba completa en cualquier plataforma.

3. ESTRUCTURA DE CODIGO

3.1 ORGANIZACION DE PAQUETES

com.ejercicio.test
- SauceDemoPurchaseFlowTest.java: Clase con todos los casos de prueba

com.ejercicio.tasks
- Login: Accion de autenticacion
- AddProductsToCart: Accion de agregar productos
- ViewCart: Accion de visualizar carrito
- ProceedToCheckout: Accion de iniciar checkout
- FillCheckoutInformation: Accion de completar formulario
- FinishPurchase: Accion de finalizar compra
- WaitForPageLoad: Accion de esperar cargas de pagina

com.ejercicio.questions
- ShoppingCartBadge: Validacion del badge de cantidad
- CartItems: Validacion de items en carrito
- OrderConfirmation: Validacion de mensaje de confirmacion

com.ejercicio.ui
- LoginPage: Elementos del formulario de login
- ProductsPage: Elementos de la pagina de productos
- CartPage: Elementos de la pagina del carrito
- CheckoutPage: Elementos de la pagina de checkout
- CheckoutCompletePage: Elementos de la pagina de confirmacion


3.2 REUTILIZABILIDAD DEL CODIGO

Las Tasks se pueden reutilizar en multiples tests y escenarios.
Las Questions permiten validaciones especificas sin duplicar codigo.
La separacion UI-Logic facilita cambios de elementos sin afectar tests.
